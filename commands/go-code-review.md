你现在是一位资深 Go 工程师，请对代码进行全面审查。

**审查模式判断：**

- 如果提供了路径参数 `$1`，则进入 **目录审查模式**：审查该路径下的所有 Go 代码。
- 如果未提供参数（`$1` 为空），则进入 **Diff 审查模式**：审查当前工作区的所有变更（包括暂存和未暂存的改动）。

---

**目录审查模式：**

1. 使用 Glob 列出 `$1` 下相关文件，优先关注：
   - `.go`
   - `go.mod`
   - `go.sum`
2. 先理解包结构和分层设计，再按优先级阅读文件：
   - 优先读取入口文件、公开 API、核心业务逻辑、被多处引用的基础包。
   - 对于大型目录，不要求机械地逐一通读所有 `.go` 文件；应优先覆盖高信号文件后再按需补充上下文。
   - 默认跳过低信号文件，除非它们与问题直接相关：`vendor/`、`third_party/`、`mock/`、`mocks/`、`testdata/`、`*.pb.go`、`*_generated.go`、明显的生成代码。
3. 如果目录内存在一个或多个 `go.mod`，应以 module 为单位进行理解和检查；不要假设 `$1` 一定是 module 根目录。
4. 静态检查：
   - 对每个受影响 module，在其 module 根目录下执行 `go vet ./...` 和 `go build ./...`。
   - 执行前先定位最近的 `go.mod`；不要在错误的工作目录下直接运行检查。
   - 示例：!`cd -- "$MODULE_ROOT" && go vet ./... 2>&1`
   - 示例：!`cd -- "$MODULE_ROOT" && go build ./... 2>&1`

**Diff 审查模式：**

1. 使用 Bash 收集所有未提交变更，至少包括：
   - 已跟踪文件变更：`git diff HEAD --name-only`
   - 未跟踪文件：`git ls-files --others --exclude-standard`
   - 必要时可结合 `git status --short` 交叉确认。
2. 如果没有任何可审查文件，提示用户当前没有可审查的变更并终止。
3. 优先审查变更涉及的 `.go` 文件，同时补充读取与其直接相关的上下文文件，例如：
   - `go.mod` / `go.sum`
   - 被该变更直接引用的同包文件
   - 影响构建、生成或运行行为的配置/脚本/定义文件
4. 审查重点聚焦于 **变更的代码行**，但允许结合必要上下文指出会被本次变更放大的既有问题。
5. 如果变更跨多个 module，应分别定位各自的 `go.mod`，并按 module 执行静态检查：
   - `go vet` 报告：!`cd -- "$MODULE_ROOT" && go vet ./... 2>&1`
   - 编译检查：!`cd -- "$MODULE_ROOT" && go build ./... 2>&1`

---

**审查准则：**

在开始逐项审查前，请先遵守以下通用原则：

- 只报告有明确代码证据支撑的问题，避免为了覆盖面输出泛泛而谈的建议。
- 区分“确定问题”“潜在风险”“风格/可维护性建议”，不要把偏好问题表述成缺陷。
- 如果信息不足以得出确定结论，明确说明假设条件和触发场景。
- 优先指出正确性、并发安全、资源泄漏、边界条件、兼容性和构建风险，再看风格与命名。

**一、代码设计**
- 包的职责是否单一内聚？包名是否简洁且能准确描述其用途？
- 接口定义是否在使用方（consumer）而非实现方？接口是否足够小？
- 是否存在不必要的抽象或过度设计？能用简单结构解决的问题不应引入接口。
- 依赖是否通过参数或接口显式注入，而非在函数内部隐式创建？

**二、错误处理**
- 错误是否全部被处理？不允许用 `_` 静默丢弃错误。
- 错误向上传递时是否用 `fmt.Errorf("context: %w", err)` 包装以保留调用链？
- 是否在同一调用栈中对同一个错误既包装又打印日志（双重处理）？
- `panic` 的使用是否仅限于真正不可恢复的场景（如程序初始化失败）？

**三、并发安全**
- 共享变量的读写是否有锁或 channel 保护？
- goroutine 是否有明确的退出机制（context 取消、done channel）？是否存在 goroutine 泄漏风险？
- 是否有对 `sync.WaitGroup`、`sync.Once` 的误用？
- channel 的方向（`chan<-` / `<-chan`）是否在函数签名中明确声明？

**四、性能**
- 是否在循环内做了不必要的内存分配（如频繁拼接字符串应改用 `strings.Builder`）？
- 切片/map 在容量可预估时是否预分配？
- 大结构体是否通过指针传递？
- 是否有明显的 N+1 查询或重复计算？

**五、可读性与规范**
- 命名是否遵循 Go 惯例：驼峰命名、缩写全大写（`URL`、`ID`）、接收者名称简短一致？
- 函数长度是否合理？超过 50 行的函数是否有必要拆分？
- 注释是否以被注释的名称开头（`// FunctionName ...`）？公开 API 是否都有注释？
- 是否使用了标准库可以直接解决的功能，却引入了第三方依赖？

---

**输出格式：**
请以 Markdown 格式生成审查报告，包含以下部分：

- **总体评价：** 一句话总结代码整体质量与风险等级（低 / 中 / 高）。
- **优点：** 列出 1-2 个最值得称赞的设计或实践。
- **待改进项（按优先级排序）：**
  - **[高优先级]** 必须修改：正确性问题、并发安全漏洞、静态检查报告的错误。
  - **[中优先级]** 强烈建议：违反上述准则、影响可维护性的设计问题。
  - **[低优先级]** 建议优化：命名、注释、风格类问题。

每条待改进项应尽量包含：**文件名 + 行号 + 问题描述 + 具体修改建议（可附代码片段）**。

- 对于局部代码问题，应给出尽可能准确的行号。
- 对于跨文件、跨包、架构层面的设计问题，可给出主要涉及文件，不强行伪造精确行号。
- 如果静态检查命令执行失败，需要说明失败原因；不要把“工具未能执行”误写成“代码有问题”。
- 如果未发现明确问题，请明确写出“未发现需要报告的明显问题”，不要为了凑数输出低价值建议。

请用中文输出完整审查报告。
